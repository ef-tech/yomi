## Problem Statement

開発中・調査中に、カレントディレクトリ配下の Markdown ファイル群をブラウザで横断的に閲覧したい。VSCode のプレビューや GitHub の表示に頼ると、複数ファイルを行き来しながら全体像を掴むのが手間。tree とプレビューを並べた専用ビューアがあれば、ドキュメント群を「読む」体験が連続する。

## What Makes This Cool

- `cd <ドキュメント置き場> && yomi` の 2 ステップで、その場で 2 ペインのドキュメントブラウザが立ち上がる。
- 配布物・依存・設定ファイルなし。Bun が入っていれば動く。
- Mermaid 図がインラインで描画される（手書きの図 → そのままブラウザで可視化）。
- コマンド名 `yomi`（読み）= Markdown を「読む」という機能と意味が直結。

## Constraints

- **個人利用** — 認証・権限管理・公開デプロイは不要。localhost バインドのみ。
- **Bun 前提** — ユーザーの環境に Bun がインストール済み。
- **再帰スキャン** — カレントディレクトリ配下を再帰的に走査。`node_modules` `.git` `dist` 等は除外。
- **2 ペイン** — 左：ファイルツリー、右：選択した md の内容＋HTML プレビュー。

## Premises (合意済み)

1. Web サーバー方式が最適（エディタ拡張・静的 HTML 出力では 2 ペイン UI 要件を満たせない）。
2. ファイル数は数千レベル想定で十分。1 万超えのリポジトリは想定外。
3. Mermaid のレンダリングはクライアント側 `mermaid.js` で行う。サーバー側は fenced code を専用クラスの `<pre>` に変換するだけ。
4. ファイル変更時のライブリロードは MVP に含める（文章ドリブンの体験が大きく良くなるため）。

## Approaches Considered

### Approach A: Bun 単一 TS（採用）

- 構成: `bin/yomi.ts` ひとつ + 静的 HTML テンプレート 1 つ。
- サーバー: `Bun.serve()` 組み込み HTTP。
- ファイル収集: `Bun.Glob("**/*.md")` で再帰列挙、除外パターン適用。
- Markdown レンダリング: `marked`（軽量、Mermaid フェンスを `<pre class="mermaid">` に変換するカスタム renderer 追加）。
- フロント: 単一 HTML に最低限の JS（fetch でファイル一覧／内容取得、ツリー描画、`mermaid.run()` 呼び出し）。
- 配布: `bun install -g .` または `bun link`。`package.json` の `bin` フィールドに `yomi` 登録。

**Effort:** S / **Risk:** Low
**Reuses:** marked（OSS）、mermaid（CDN または bundled）

Pros:
- 起動時間 < 100ms。Bun の組み込み機能だけで完結。
- ファイル数 4〜5。読み返しやすい。
- 個人利用の文脈に過不足なし。

Cons:
- Bun 環境が前提。他人に共有するときは Bun のインストール案内が必要。
- TypeScript の型定義 (`@types/marked`) を別途入れる必要あり。

### Approach B: Node + Express

- 構成: `express` + `marked` + `chokidar`（ライブリロード時用）。
- 配布: `npm install -g`。

**Effort:** M / **Risk:** Low

Pros:
- エコシステムが厚い、知見豊富。
- 他人にも共有しやすい（Node はほぼ全員入っている）。

Cons:
- `node_modules` が肥大（express だけで数十パッケージ）。
- 起動 200〜400ms 程度、Bun より遅い。
- 個人利用に対しオーバーエンジニアリング気味。

### Approach C: Go 単一バイナリ

- 構成: `net/http` + `goldmark` + `embed`。HTML/JS/CSS を埋め込み。
- 配布: クロスコンパイルで macOS/Linux/Windows のバイナリを GitHub Releases に置く。

**Effort:** M / **Risk:** Med

Pros:
- 配布最強。ランタイム不要。
- 起動高速、メモリ効率良。

Cons:
- 言語が変わる。今後の保守コストが上がる可能性。
- 個人利用 1 環境にバイナリ配布の利点が薄い。

## Recommended Approach

**Approach A (Bun 単一 TS)** を採用。
理由：個人利用 × Bun 既存環境という条件で、ファイル数最小・起動最速・実装最短の組み合わせ。Mermaid 対応もクライアント側 JS で完結するため、サーバー実装は marked のカスタム renderer 1 行追加で済む。

## Architecture Sketch

```
yomi/
├── package.json          # bin: { "yomi": "./bin/yomi.ts" }
├── bin/
│   └── yomi.ts           # CLI エントリ + サーバー + ブラウザ自動オープン
├── src/
│   ├── server.ts         # Bun.serve() ロジック (HTTP + WebSocket)
│   ├── scanner.ts        # 再帰スキャン + 除外
│   ├── watcher.ts        # Bun.watch で md 変更検知 → WebSocket broadcast
│   └── renderer.ts       # marked + Mermaid renderer
├── public/
│   ├── index.html        # 2 ペイン UI
│   ├── app.js            # ツリー描画・プレビュー・mermaid.run()・WS リロード
│   └── styles.css        # github-markdown-css 取込 + prefers-color-scheme
└── README.md
```

### サーバー API

| エンドポイント         | 役割                                                   |
|------------------------|--------------------------------------------------------|
| `GET /`                | `index.html` を返す                                    |
| `GET /api/tree`        | カレントディレクトリ配下の md ツリー JSON              |
| `GET /api/file?path=…` | 指定 md ファイルの raw 内容と HTML を返す              |
| `GET /assets/*`        | `public/` 以下の静的アセット                           |
| `WS  /ws`              | ファイル変更通知。サーバーが `{type:"changed",path}` を push |

### 除外パターン（デフォルト）

`node_modules`, `.git`, `dist`, `build`, `.next`, `.cache`, `coverage`, `vendor`

### Mermaid 対応の実装

```typescript
// renderer.ts
import { marked } from "marked";

const renderer = new marked.Renderer();
const originalCode = renderer.code.bind(renderer);
renderer.code = (code, language) => {
  if (language === "mermaid") {
    return `<pre class="mermaid">${escapeHtml(code)}</pre>`;
  }
  return originalCode(code, language);
};
```

クライアント側 `app.js` で：
```javascript
import mermaid from "https://cdn.jsdelivr.net/npm/mermaid@11/dist/mermaid.esm.min.mjs";
mermaid.initialize({ startOnLoad: false });
// プレビュー描画後に
await mermaid.run({ nodes: document.querySelectorAll("pre.mermaid") });
```

## Security Considerations (個人利用前提でも最低限)

- **localhost (`127.0.0.1`) バインドのみ** — `0.0.0.0` で外部公開しない。
- **パストラバーサル防止** — `/api/file?path=` は `path.resolve()` してカレントディレクトリ配下かを検証。`..` を含むパスは拒否。
- **HTML サニタイズ** — 信頼できる自分のファイル前提だが、`marked` の `sanitize` オプションは無効でよい（Mermaid 等の複雑な HTML を許容するため）。代わりに「外部から渡されたパスをロードしない」設計で防御。

## Resolved Decisions

| 論点 | 決定 |
|------|------|
| ポート | **自動探索**。3939 を起点に空きポートを探す。起動時にコンソールへ URL 表示 |
| ブラウザ自動オープン | **デフォルト ON**。`--no-open` で抑止できる |
| ファイルウォッチ | **MVP に含める**。`Bun.watch` で md 変更を検知 → WebSocket でブラウザに通知 → 該当ファイル表示中ならプレビュー差し替え |
| CSS テーマ | **GitHub 風 + システム追従**。`github-markdown-css` を取り込み、`prefers-color-scheme` でダーク/ライト自動切替 |

### CLI フラグまとめ

```
yomi [options]
  --port <n>      ポートを指定（デフォルト: 3939 から自動探索）
  --no-open       ブラウザを自動で開かない
  --host <addr>   バインドアドレス（デフォルト: 127.0.0.1）
  --help, -h      ヘルプ
```

## Success Criteria

- カレントディレクトリで `yomi` 実行 → 1 秒以内にブラウザが起動し、左ツリーに md 一覧、右ペインに最初のファイルがレンダリングされる。
- ` ```mermaid ` ブロックがブラウザ上で図として描画される。
- ファイル数 1000 程度の規模で UI が固まらない。

## Next Steps（実装順）

1. `package.json` 作成、`bin: { "yomi": "./bin/yomi.ts" }` 設定、`marked` を依存追加。
2. `bin/yomi.ts` で CLI 引数パース（`--port`, `--no-open`, `--host`, `--help`）+ `Bun.serve()` 起動、`/` で hello world。空きポート探索ロジックを実装。
3. `src/scanner.ts` 実装、`/api/tree` を返せるようにする（除外パターン適用）。
4. `src/renderer.ts` で marked + Mermaid renderer、`/api/file` を実装。
5. `public/index.html` + `app.js` で 2 ペイン UI。`github-markdown-css` の取込と `prefers-color-scheme` 対応。最初のファイルを自動選択。
6. クライアント側に Mermaid.js を追加し、プレビュー後に `mermaid.run()`。
7. `src/watcher.ts` で `Bun.watch` 実装、`/ws` で WebSocket。ブラウザ側に再フェッチロジック追加。
8. パストラバーサル検証、localhost バインド固定。`--no-open` 以外ではブラウザ自動オープン。
9. README.md（インストール手順、除外パターンのカスタマイズ方法、CLI オプション）。
10. 初期コミット & push to `ef-tech/yomi`。

## What I noticed about how you think

- 「nokogiri みたいに日本語で且つ覚えれそうなコマンド名」という言い回しから、**ツールの名前と機能の意味的な結びつき** に強いこだわりがある。`mdv` を仮置きで提示しつつ「もっと良いのが欲しい」と明言したのは健全な taste。
- 「個人的な利用」と最初に断言した上で要件をミニマルに絞った。スコープを広げて公開ツールにしたいという誘惑に乗らない判断は良い設計者の特徴。
- Mermaid 対応の指摘 — 純粋な Markdown ビューアを超えて「自分が書く図表のレンダリング」まで視野に入っている。普段のドキュメント作成スタイルが透ける質問。
